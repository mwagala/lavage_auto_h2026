Roadmap Fonctionnalites - Stack Actuelle

Documentation par etape pour la version Flask / Jinja templates existants / MySQL, sans migration vers un framework frontend separe, avec les briques Redis, Celery, Flask-SocketIO, SendGrid, Twilio, Stripe, WeasyPrint et S3. Chaque page resume les stacks a implementer, leur role dans ce projet et les gestes d'installation a suivre sans entrer dans le code.

1. Fondations 2. 2FA 3. Notifications 4. Paiement en ligne 5. Facture PDF 6. Tracking live 7. Rappels 24h / 2h 8. Entente de service 9. Photos et documents cloud 10. QA et mise en prod

Etape 1

Fondations

Objectif: preparer l'application Flask existante pour les traitements fiables en arriere-plan, les evenements metier, l'idempotence, l'audit, les erreurs lisibles et une observation minimale avant d'ajouter les fonctionnalites plus sensibles.

Stacks a implementer

Redis

A quoi ca sert. Redis est une base en memoire tres rapide, souvent utilisee comme file de messages, cache et support de verrous courts.

Dans ce projet. Il sert de broker pour Celery et de support technique pour les taches asynchrones et, plus tard, SocketIO multi-processus.

  1. Installer Redis en local ou via un conteneur de developpement.
  2. Ajouter les parametres de connexion dans la configuration du projet.
  3. Verifier que l'application peut se connecter au service au demarrage.
  4. Prevoir une configuration differente pour developpement, test et production.

Celery

A quoi ca sert. Celery execute des taches longues ou non critiques hors de la requete HTTP principale.

Dans ce projet. Il evitera de bloquer les clients pendant les emails, SMS, rappels, generation PDF et traitements d'outbox.

  1. Creer une application Celery reliee a la configuration Flask.
  2. Definir un dossier de taches dedie aux traitements asynchrones.
  3. Lancer un worker separe de l'application Flask.
  4. Tester une tache simple avant de brancher les cas metier.

Outbox MySQL

A quoi ca sert. Le pattern outbox enregistre les evenements dans la meme transaction que les donnees metier.

Dans ce projet. Il garantit qu'une reservation creee, modifiee ou annulee declenche un evenement meme si un worker est temporairement indisponible.

  1. Ajouter une table d'evenements avec type, payload, statut, nombre de tentatives et dates.
  2. Inserer un evenement dans la meme transaction que l'action metier.
  3. Faire lire les evenements en attente par un consumer Celery.
  4. Associer une cle d'idempotence pour eviter le traitement multiple du meme evenement.
  5. Marquer chaque evenement comme traite ou en erreur avec un message exploitable.

Flask/Jinja templates existants

A quoi ca sert. Flask sert le backend et Jinja rend les pages HTML cote serveur avec un peu de JavaScript progressif au besoin.

Dans ce projet. Le projet reste en Flask: on enrichit les routes, services, repositories et templates existants au lieu de migrer vers un framework frontend separe.

  1. Conserver les blueprints Flask deja presents.
  2. Ajouter les nouvelles pages dans les dossiers de templates existants.
  3. Utiliser JavaScript seulement pour les interactions necessaires, comme Stripe, SocketIO ou upload S3.
  4. Eviter une architecture SPA tant que le besoin metier ne l'exige pas.

Logs et erreurs standardises

A quoi ca sert. Les logs structurent ce qui arrive dans le systeme; les erreurs standardisees rendent les reponses API previsibles.

Dans ce projet. Les modules routes, services et repositories pourront renvoyer des erreurs coherentes, utiles pour le frontend et les tests.

  1. Definir un format commun de reponse pour succes et erreur.
  2. Centraliser la journalisation des exceptions techniques.
  3. Distinguer erreurs utilisateur, erreurs metier et erreurs internes.
  4. Ajouter un identifiant de correlation pour suivre une requete dans les logs.

Idempotence et journal d'audit

A quoi ca sert. L'idempotence evite les doublons quand une action est rejouee; l'audit garde une trace exploitable des actions sensibles.

Dans ce projet. Ces deux briques securisent Stripe, outbox, notifications, rappels, 2FA, signatures, medias et changements de reservation.

  1. Definir les actions qui doivent etre rejouables sans effet double.
  2. Ajouter des cles uniques ou verrous courts pour les operations critiques.
  3. Creer une table d'audit avec acteur, action, ressource, resultat, IP et correlation id.
  4. Brancher l'audit dans les services metier plutot que dans les templates.

Etape 2

2FA

Objectif: transformer le login actuel en connexion en deux etapes avec TOTP, codes de recuperation et fallback SMS, tout en ne delivrant le JWT final qu'apres validation du second facteur.

Stacks a implementer

TOTP

A quoi ca sert. TOTP genere des codes temporaires compatibles avec des applications comme Google Authenticator ou Microsoft Authenticator.

Dans ce projet. Les clients et prestataires protegent leur compte sans dependre d'un SMS a chaque connexion.

  1. Ajouter un secret TOTP par utilisateur active pour le 2FA.
  2. Prevoir un flux d'enrolement avec QR code ou cle manuelle.
  3. Verifier un code avant d'activer definitivement le 2FA.
  4. Limiter les tentatives pour eviter les attaques par essais repetes.

Recovery codes

A quoi ca sert. Les codes de recuperation permettent de se connecter si l'utilisateur perd son application d'authentification.

Dans ce projet. Ils evitent de bloquer un client ou prestataire avant une reservation importante.

  1. Generer une liste courte de codes a usage unique.
  2. Stocker uniquement des versions hachees des codes.
  3. Afficher les codes une seule fois apres activation.
  4. Marquer un code comme consomme apres utilisation.

Twilio OTP SMS

A quoi ca sert. Twilio permet d'envoyer des SMS transactionnels contenant un code temporaire.

Dans ce projet. Il sert de secours quand le TOTP n'est pas disponible ou pendant la transition vers le 2FA.

  1. Valider que le numero de telephone de l'utilisateur est fiable.
  2. Creer un challenge OTP avec une expiration courte.
  3. Envoyer le code par SMS via Twilio.
  4. Bloquer la reutilisation du code apres validation ou expiration.

Flask-JWT-Extended

A quoi ca sert. Cette extension gere les jetons JWT utilises pour authentifier les appels API.

Dans ce projet. Le login ne doit plus emettre le token final tant que le second facteur n'est pas valide.

  1. Retourner un etat temporaire apres email et mot de passe valides.
  2. Associer cet etat a une session de login courte duree.
  3. Emettre le JWT final seulement apres validation 2FA.
  4. Ajouter des tests pour les chemins TOTP, SMS et recovery code.

Audit securite

A quoi ca sert. Un audit securite conserve les tentatives importantes sans stocker les secrets ou les codes en clair.

Dans ce projet. Il permet de retracer activation 2FA, login reussi, echec TOTP, fallback SMS et usage de recovery code.

  1. Identifier les evenements de securite a journaliser.
  2. Ne jamais stocker le code TOTP, OTP SMS ou recovery code en clair.
  3. Associer chaque trace a l'utilisateur, au role, a l'adresse IP et au correlation id.
  4. Prevoir une consultation interne limitee aux besoins de support et securite.

Etape 3

Notifications

Objectif: envoyer les messages email et SMS pour les evenements de reservation sans ralentir les actions utilisateur: confirmation, annulation, modification, en route, en cours et termine.

Stacks a implementer

Celery + Redis

A quoi ca sert. Cette combinaison permet de mettre les notifications en file et de les traiter hors requete.

Dans ce projet. Une reservation peut etre confirmee immediatement meme si l'envoi email ou SMS prend du temps.

  1. Declarer une tache de livraison de notification.
  2. Brancher la tache sur le consumer outbox.
  3. Journaliser chaque tentative d'envoi.
  4. Prevoir des relances en cas d'erreur temporaire.

SendGrid

A quoi ca sert. SendGrid envoie des emails transactionnels avec suivi de statut et templates.

Dans ce projet. Il sert aux confirmations, rappels, changements de reservation et documents importants.

  1. Creer une cle API limitee a l'envoi d'emails.
  2. Definir les expediteurs autorises.
  3. Preparer les templates de messages par type d'evenement.
  4. Tester les emails avec une adresse de developpement avant production.

Twilio SMS

A quoi ca sert. Twilio envoie des SMS courts et urgents aux utilisateurs.

Dans ce projet. Il complete l'email pour les changements sensibles, comme prestataire en route ou annulation proche.

  1. Verifier le format des numeros de telephone.
  2. Limiter les SMS aux evenements utiles pour eviter le bruit.
  3. Centraliser le texte des messages.
  4. Enregistrer le statut d'envoi pour le support.

Outbox d'evenements

A quoi ca sert. L'outbox transporte les evenements metier de facon fiable vers les notifications.

Dans ce projet. Elle evite qu'une reservation soit modifiee sans que les utilisateurs soient informes.

  1. Associer chaque changement de statut a un type d'evenement clair.
  2. Mettre les donnees minimales necessaires dans le payload.
  3. Eviter d'envoyer le meme evenement deux fois.
  4. Conserver l'historique pour audit et debogage.

Idempotence de livraison

A quoi ca sert. Une livraison idempotente garantit qu'un meme evenement rejoue ne produit pas plusieurs messages identiques.

Dans ce projet. Elle evite les doubles emails ou doubles SMS si Celery relance une tache ou si l'outbox est rejouee.

  1. Construire une cle unique par evenement, destinataire et canal.
  2. Verifier cette cle avant de creer un message.
  3. Conserver le statut de livraison pour les relances controlees.
  4. Distinguer relance technique et nouvel evenement metier.

Etape 4

Paiement en ligne

Objectif: permettre au client de payer une reservation avec Stripe Payment Intent, verifier les webhooks signes et rapprocher proprement reservation, facture et transaction.

Stacks a implementer

Stripe Payment Intent

A quoi ca sert. Payment Intent represente une tentative de paiement et gere les etats modernes comme authentification bancaire et paiement confirme.

Dans ce projet. Chaque reservation payable aura un intent Stripe lie au montant facture.

  1. Creer un compte Stripe de test et recuperer les cles API.
  2. Calculer le montant cote serveur a partir de la facture.
  3. Creer un Payment Intent pour la reservation.
  4. Retourner au frontend uniquement les donnees necessaires au paiement.

Webhooks signes Stripe

A quoi ca sert. Les webhooks informent le backend du resultat final du paiement, avec une signature pour verifier l'origine.

Dans ce projet. Ils deviennent la source fiable pour marquer une transaction comme payee ou echouee.

  1. Creer un endpoint public dedie aux webhooks Stripe.
  2. Configurer le secret de signature dans l'environnement.
  3. Verifier la signature avant toute mise a jour.
  4. Rendre le traitement idempotent pour accepter les repetitions Stripe.

MySQL rapprochement

A quoi ca sert. Le rapprochement relie le statut externe du paiement aux donnees internes.

Dans ce projet. Il synchronise reservation, facture et transaction pour eviter les factures payees sans trace ou les reservations marquees payees trop tot.

  1. Ajouter une table des transactions de paiement.
  2. Enregistrer l'identifiant Payment Intent et le montant attendu.
  3. Mettre a jour la transaction via webhook signe.
  4. Recalculer ou valider l'etat de paiement de la facture.

Frontend paiement

A quoi ca sert. Le frontend collecte les informations de paiement avec les composants securises Stripe.

Dans ce projet. Il permet au client de payer depuis son tableau de bord ou la page de reservation.

  1. Afficher le recapitulatif du montant avant le paiement.
  2. Demander au backend la creation du Payment Intent.
  3. Confirmer le paiement avec Stripe cote navigateur.
  4. Afficher un etat clair: en attente, paye, refuse ou a reessayer.

Audit paiement et idempotence

A quoi ca sert. L'audit conserve les transitions sensibles; l'idempotence protege contre les webhooks repetes.

Dans ce projet. Une transaction Stripe repetee ne doit jamais payer deux fois une facture ni envoyer deux confirmations.

  1. Enregistrer chaque evenement Stripe recu avec son identifiant externe.
  2. Refuser ou ignorer proprement les evenements deja traites.
  3. Journaliser les passages en paye, echoue, rembourse ou en litige.
  4. Inclure reservation, facture, transaction et correlation id dans l'audit.

Etape 5

Facture PDF

Objectif: generer une facture detaillee, telechargeable de facon securisee, a partir des donnees de reservation, services, taxes et transaction de paiement.

Stacks a implementer

WeasyPrint

A quoi ca sert. WeasyPrint transforme une page HTML et CSS en fichier PDF propre.

Dans ce projet. Il permet de reutiliser un template de facture lisible et de produire un PDF officiel pour le client.

  1. Preparer un template HTML de facture avec les donnees obligatoires.
  2. Definir une feuille de style adaptee au format PDF.
  3. Tester la generation sur une facture simple puis detaillee.
  4. Verifier le rendu visuel avant de l'exposer au client.

Endpoint Flask securise

A quoi ca sert. Un endpoint controle qui peut telecharger une facture et dans quelles conditions.

Dans ce projet. Il evite qu'un client accede a la facture d'un autre client ou a une facture non prete.

  1. Verifier l'identite JWT de l'utilisateur.
  2. Verifier que l'utilisateur est lie a la reservation.
  3. Generer le PDF a la demande ou recuperer une version existante.
  4. Retourner un telechargement ou une URL temporaire securisee.

MySQL facture detaillee

A quoi ca sert. La base fournit les lignes de services, montants, taxes et paiements a afficher.

Dans ce projet. Les donnees viennent deja de Reservations, Reservation_Services, Services et Factures.

  1. Valider que les montants de facture sont complets.
  2. Ajouter les champs manquants si le PDF doit afficher taxes, numero ou statut.
  3. Charger toutes les lignes de service en une requete controlee.
  4. Conserver une trace de generation si le PDF devient officiel.

S3 pour conservation

A quoi ca sert. S3 stocke des fichiers de facon durable hors de la base de donnees.

Dans ce projet. Il peut conserver les PDF generes et servir des liens temporaires au client.
